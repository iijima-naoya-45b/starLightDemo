## ⚠️ 危機管理とBCP

危機管理とBCPは「紙の計画」ではなく、**システムに組み込まれた自律的な回復力**です。
ゴールは「計画書を作ること」ではなく、**手動介入なしでコア業務が継続するアーキテクチャ**を実現することです。

### 🎯 なぜ重要なのか（攻めの事業戦略）

- **営業の武器**: 監査・審査で強い（SLAや復旧体制が契約を通す）
- **顧客の継続率**: 障害後の離脱を防げる（復旧速度が満足度を左右する）
- **開発速度**: リリースを攻められる（安全弁があると大胆に動ける）
- **採用・評価**: 強い組織の証明になる（運用力は技術力そのもの）

## 1. これだけは死守する「コア業務」

「全部やる」ではなく、**事業が回り続ける最低ライン**を明文化します。

**コア業務の例（自社に合わせて調整）:**

- **売上直結**: 課金・決済・請求書発行
- **サービス継続**: 認証、主要API、ユーザーデータ参照
- **顧客接点**: 問い合わせ受付、ステータスページ
- **法的リスク回避**: 個人情報の保全、監査ログの保全

## 2. コア業務の特定プロセス（これだけはやる）

1. **価値の流れを分解する**  
   「ユーザー登録 → 認証 → 主要機能 → 課金」までを一枚の流れで可視化する。

2. **最低継続ラインを決める**  
   例: 「閲覧は維持、更新系は停止」「決済は維持、分析は停止」。

3. **RTO/RPOを決める**  
   例: 「決済はRTO 30分 / RPO 5分」「問い合わせはRTO 2時間 / RPO 24時間」。

4. **依存関係を固定する**  
   例: 「認証が落ちたら閲覧をゲストに切り替える」「決済は外部代替へ切替」。

5. **オーナーを決める**  
   「この領域が止まったら誰が決断するか」を明文化する。

## 3. 危機の種類と具体対策（技術で語る）

### 3.1 自然災害（DC停止・拠点停止）

- **冗長化**: マルチAZ / マルチリージョン構成
- **バックアップ**: 3-2-1バックアップ、PITR、定期リストア検証
- **フェイルオーバー**: 自動切替手順のRunbook化
- **通信経路**: 回線の二重化、DNS切替手順

### 3.2 人的災害（不正・誤操作・人員不足）

- **権限管理**: 最小権限、ブレークグラス権限、操作承認フロー
- **監査ログ**: 全操作の改ざん不可能なログ保存
- **変更管理**: IaC / GitOpsで変更を記録・再現可能に
- **引き継ぎ**: キーマン離脱を想定したRunbookと権限棚卸し

### 3.3 技術的災害（障害・攻撃・データ損失）

- **可用性**: ロードバランサー、オートスケール、セルフヒーリング
- **データ保全**: 書込みログ、スナップショット、暗号化
- **監視**: SLO監視、アラートの閾値設計、インシデント自動起票
- **セキュリティ**: WAF、EDR、脆弱性スキャン、秘密情報の管理

### 3.4 経営危機（資金・信用・法務）

- **キャッシュ耐性**: ランウェイの可視化、コストの固定化回避
- **法務対応**: 契約・監査の棚卸し、責任範囲の明文化
- **信用維持**: ステータスページ・広報テンプレートの用意

## 4. 危機管理の実行フロー（手順で語る）

1. **検知**: 監視アラート、ログ異常、顧客問い合わせの自動集約  
2. **初動**: 影響範囲の特定、優先度決定、復旧方針の決定  
3. **封じ込め**: 変更停止、アクセス制限、ロールバック  
4. **復旧**: フェイルオーバー、リストア、代替運用  
5. **事後分析**: 再発原因、恒久対応、改善の反映  

## 5. BCPの実践（紙ではなくシステム）

- **定期リストア演習**: バックアップから30分以内で復旧できるか
- **縮退運転**: 機能を段階的に落とし、コア業務を守る
- **代替手段**: 手動請求、CSV運用、代替コミュニケーション
- **訓練=自動テスト**: 定期的に本番の一部を意図的に落とし、復旧速度を計測する

### 自律的な回復力（Engineering Resilience）の構成

「BCP担当が守る計画」ではなく、**システム自体が回復する構造**にします。

- **自己回復**: ヘルスチェックと自動再起動で復旧する
- **自己防御**: フェイルオーバーとレート制限で被害を封じる
- **自己検証**: 障害注入（カオス実験）で復旧速度を常時測定する
- **自己修復**: バックアップと自動リストアで復旧する

### 訓練は「自動テストの一部」

- **本番の一部を落とす**: 対象は影響の小さい領域から段階的に拡大する
- **復旧速度を記録する**: RTO/RPOを満たさなければ改善対象にする
- **アラートと復旧の連動**: 手動対応ではなく自動復旧が成功したかを評価する

### 体制: SRE責任を全員に

- **BCP担当を廃止**: 「誰かが守る」体制をやめる
- **全員が信頼性に責任を持つ**: 各チームが自分のサービスのSLOを持つ
- **レビューに信頼性を組み込む**: 変更時に可用性・復旧性を必ず確認する

## 6. まとめ

危機管理とBCPは「守り」ではなく「攻めの事業戦略」です。
紙の計画を捨て、**自律的に回復するアーキテクチャ**として実装し、信頼を売り、顧客を守り、開発を加速させます。
