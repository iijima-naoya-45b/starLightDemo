## Qiita vs Zenn：エンジニアのための「戦場」選び

「エンジニアならどっちに書くべきか」という論争に、  
**「情報の鮮度」と「ストックの価値」**という観点から終止符を打ちます。

Qiitaは「街の巨大な掲示板」、Zennは「洗練された個人の書斎」。  
この性質の違いを理解せずに投稿するのは、タキシードで居酒屋に行くようなものです。

### 1. 概念の違い：コミュニティか、パブリッシングか

| 項目 | Qiita（キータ） | Zenn（ゼン） |
|------|----------------|--------------|
| メタファー | 技術のお祭り会場 | 技術者のナレッジベース |
| 文化 | 集合知。みんなでワイワイ知識を持ち寄る。 | 個人のアウトプット。質と体系性を重視。 |
| 強み | 圧倒的なSEO（検索の強さ） | MarkdownとGitHub連携の心地よさ |

### 2. Qiita：圧倒的な拡散力と「お祭り」感

Qiitaは、日本最大のエンジニアコミュニティとしての**「数の暴力（良い意味で）」**が最大の武器です。

- **SEOの化け物**: エラーメッセージで検索して最初に出てくるのは、今も昔もQiita。  
- **「いいね」の爆発力**: 記事がトレンドに乗った時の拡散スピードは凄まじい。  
- **初心者への寛容さ**: 「やってみた」「詰まった」が受け入れられる土壌がある。  

### 3. Zenn：知性を形にする「エンジニア・ファースト」

Zennは、後発ゆえにエンジニアが「本当に欲しかった執筆体験」を追求しています。

- **GitHub連携は正義**: ローカルで書き、git pushで公開する体験。  
- **Mermaid対応で図解が捗る**: 設計や構成の解説に強い。  
- **本（Book）で体系化・マネタイズ**: 断片ではなく「資産」を作れる。  

### 4. どっちに何を書くべきか（マサカリ的判断基準）

「どっちがいい」ではなく、**「情報の性質」**で使い分けろ。

✅ **Qiitaに投下すべきネタ**

- **特定のバグの解決策**: 検索に引っかかって誰かを救うことが目的。  
- **ポエム・キャリア論**: 感情を揺らし、議論を呼びたい時。  
- **やってみた系速報**: 新機能を早く触って、お祭り騒ぎに参加したい時。  

✅ **Zennに刻むべきネタ**

- **体系的な技術解説**: 1万字を超える深く重厚な内容。  
- **図解メインの設計論**: Mermaidで視覚的に理解させたい時。  
- **開発ログ（スクラップ）**: 完成品ではなく、試行錯誤の過程を残したい時。  

### 5. 比較マトリックス：機能の差は「思想」の差

| 機能 | Qiita | Zenn |
|------|-------|------|
| 図解（Mermaid） | ❌（画像のみ） | ✅（コードで書ける） |
| 記事管理 | Webエディタ中心 | Localエディタ + GitHub |
| 収益化 | ❌（ブランド向上のみ） | ✅（本の販売、バッジ） |
| 下書き共有 | ✅（限定共有URL） | ✅（GitHubのPRで管理可） |

### 結論：あなたの「狙い」はどこにある？

- 「とにかく多くの人の目に触れたい！トレンド入りしてドヤりたい！」  
  👉 **Qiitaへ行け**

- 「知見を美しく体系化し、長く残る資産にしたい！」  
  👉 **Zennで書け**
